# Device Trust and Admin Controls Architecture

Extends architecture coverage based on the reference-app gap review.

## Purpose

This document defines the anti-abuse design that prevents one physical phone from being used to rotate through multiple student accounts for attendance marking.

## Reset Implementation Snapshot

- student signup binds the first attendance phone
- replacement-device attempts now create pending recovery state instead of silently taking over
- admin student support and device recovery are now explicit, audited web workflows
- remaining work in this area is attestation hardening and real-device validation, not policy ambiguity

## Threat Model

The main abuse scenarios are:

- student A logs in, marks attendance, logs out
- student B logs in on the same phone and marks attendance
- stolen phone continues to be used after ownership change
- app reinstall attempts to bypass lightweight client-only locks

The solution must be stronger than UI-only logout restrictions.

## Important Design Decision

Do not rely on MAC address as the device identity.

Reason:

- modern mobile operating systems do not reliably expose stable MAC addresses to apps
- MAC addresses may be randomized
- they are not appropriate as the main security primitive for this product

Instead, use a trusted-device architecture built from app-scoped identifiers and attestation.

## Device Identity Components

Each app installation should generate or collect:

- `install_id` generated by the app
- device key pair stored in secure keystore / keychain when possible
- platform metadata such as OS, app version, device model
- attestation proof where supported

### Platform Attestation

Recommended options:

- Android: Play Integrity or equivalent attestation
- Apple: App Attest / DeviceCheck style attestation where feasible

These signals help distinguish a normal device from a tampered or cloned environment.
In the current auth foundation, client-supplied attestation metadata is stored but not treated as verified proof until this phase adds explicit server-side attestation validation.

## Binding Policy

For student attendance marking in v1, use a strict policy:

- one device installation can be actively bound to one student account
- one student account can have one active attendance-eligible device at a time

This is intentionally strict because the product goal is anti-proxy integrity.

Teacher and admin accounts can use a softer policy because they are not the high-risk attendance-marking actor.

## Binding State Model

Suggested binding statuses:

- `PENDING`
- `ACTIVE`
- `REVOKED`
- `BLOCKED`

Student-facing lifecycle states that should be derived from those records and returned to clients:

- `TRUSTED`
- `PENDING_REPLACEMENT`
- `REPLACED`
- `BLOCKED`
- `UNREGISTERED`
- `NOT_APPLICABLE`

Suggested device attestation statuses:

- `UNKNOWN`
- `PASSED`
- `FAILED`
- `NOT_SUPPORTED`

## First Registration, Login, and Binding Flow

1. the app gathers installation and device proof
2. the client sends that device metadata in student registration, login, or Google exchange payloads
3. the backend upserts the `devices` record
4. the backend checks whether the device is already bound to another active student
5. on student self-registration, the backend completes that conflict check before creating the student account
6. on student login, the backend also checks whether the student is already bound to a different active device
7. if allowed, binding becomes `ACTIVE`
8. if a second phone tries to replace an existing trusted student phone, the backend creates or reuses a `PENDING` replacement binding and blocks attendance on that new phone until admin approval
9. if not allowed, the backend rejects student registration or authentication instead of opening a new student session

Current implementation note:

- the API still exposes trusted-device state through `GET /auth/me` for support visibility
- successful student self-registration now returns both the trusted session and an onboarding hint for the classroom-join step
- dedicated support and recovery endpoints are now implemented through the admin device-support module
- `TrustedDeviceContext` now also carries a lifecycle-state field so mobile and web clients can distinguish `Trusted`, `Pending approval`, `Replaced`, `Blocked`, and `Unregistered` without inventing new heuristics

## Attendance-Time Enforcement

Every student attendance request should pass a `TrustedDeviceGuard`.

Guard checks:

- session token belongs to student
- device binding exists and is active
- request is coming from the expected install context
- binding is not revoked
- suspicious switch or reinstall pattern is not currently blocked

Only after this should QR or BLE validation continue.

This guard is now implemented and first attached to `GET /devices/trust/attendance-ready`. Future QR and BLE mark-attendance APIs must reuse the same guard before continuing into mode-specific validation.

## Reinstall and Clone Resistance

An app reinstall should not automatically be treated as the same trusted device.

Recommended handling:

- new install creates new `install_id`
- backend sees it as a new device registration
- if old binding exists, new install requires admin or recovery approval

This makes casual reinstall-based bypassing much harder.

## Security Events

The system should log events such as:

- `DEVICE_BOUND`
- `DEVICE_REVOKED`
- `ATTENDANCE_BLOCKED_UNTRUSTED_DEVICE`
- `MULTI_ACCOUNT_SAME_DEVICE_ATTEMPT`
- `SECOND_DEVICE_FOR_STUDENT_ATTEMPT`
- `REVOKED_DEVICE_USED`

The current foundation already persists:

- `DEVICE_BOUND`
- `MULTI_ACCOUNT_SAME_DEVICE_ATTEMPT`
- `SECOND_DEVICE_FOR_STUDENT_ATTEMPT`
- `REVOKED_DEVICE_USED`
- `LOGIN_RISK_DETECTED`

`security_events` can now also reference `user_device_bindings` directly through `binding_id`, which improves support correlation and later admin tooling.

These events feed support tooling and optional risk metrics.

## Admin Recovery and Delink

Admins need a dedicated support surface for legitimate recovery scenarios.

Supported support actions:

- review student account state and recent classroom context before recovery
- deactivate or archive student sign-in access with audit trail when governance requires it
- delink lost or broken phone from student
- revoke suspicious device binding
- approve a replacement device
- review recent security events for a student

Reset governance boundary:

- device recovery remains separate from classroom-student membership management
- when an admin also needs to correct classroom membership, that should happen through the shared
  classroom student CRUD surface in `apps/api/src/modules/academic/roster.service.ts`, not through
  device-support-only endpoints
- when an admin needs to archive or govern a classroom lifecycle decision, that now belongs to the
  academic-governance workspace under `/admin/semesters`, not the device-recovery desk

Recommended admin endpoints:

- `GET /admin/students`
- `GET /admin/students/:studentId`
- `POST /admin/students/:studentId/status`
- `GET /admin/device-bindings`
- `GET /admin/device-bindings/:studentId`
- `POST /admin/device-bindings/:bindingId/revoke`
- `POST /admin/device-bindings/:studentId/delink`
- `POST /admin/device-bindings/:studentId/approve-new-device`

All admin actions must create audit rows.

Reset deletion rule:

- device recovery actions revoke bindings and preserve device/binding history
- there is no normal product hard-delete path for devices or student attendance bindings
- support tooling should use `revoke`, `deregister current phone`, and `approve replacement` language while backend rows stay audit-safe

## Optional Self-Service Recovery

Later versions may support self-service recovery after secondary verification, such as:

- institutional email OTP
- admin-approved support ticket
- temporary teacher recommendation flow

For v1, admin-assisted recovery is simpler and safer.

## Teacher and Student UX Expectations

### Student UX

- clear message when device is bound successfully
- clear block when attendance is disabled on current device
- support instructions when phone is changed

Current implementation note:

- the mobile shell now has a `buildStudentAttendanceGateModel()` helper for blocked-attendance messaging
- placeholder attestation descriptions are exposed for Android and iOS until native verification adapters are wired
- the blocked-attendance UX applies only to student attendance-sensitive flows, while teacher and admin sign-in paths remain outside the strict student trust gate
- the dedicated student device-status screen now renders the lifecycle-state labels and the attendance-readiness result from `GET /devices/trust/attendance-ready`
- native Android validation now proves the trusted seeded phone can reach a device-status view showing `Attendance device state: Trusted` and `Binding status: ACTIVE`

### Teacher UX

- no heavy device-binding friction for normal teacher usage
- optional warning if a student is blocked from attendance due to device issue

### Admin UX

- searchable student and device records
- view active and historical bindings
- deregister-current-phone and approve-replacement actions with audit trail

Current implementation note:

- the first admin console is now available in the protected admin web shell at `/admin/devices`
- `/admin/devices?view=support` now uses the dedicated `/admin/students` support API so admins can search students, inspect account state, inspect recent classroom context, and apply audited `USER_STATUS_CHANGE` actions before they enter recovery
- admin device-recovery summary/detail payloads now include the current trusted phone, pending replacement phone, the latest risk event, the latest recovery action, and an API-backed recommended next step
- `/admin/devices` still focuses on guarded recovery actions over `/admin/device-bindings`
- `/admin/semesters` now owns the separate academic-governance lane for classroom archive decisions
  with history-impact review and live-session blocking
- the protected admin web session is now the primary access path for support and recovery, with manual token override kept secondary for controlled validation
- revoke, deregister-current-phone, and replacement approval now require a recorded reason plus an explicit admin acknowledgement before the action can run
- if a pending replacement exists, clearing the current phone still does not auto-trust that pending phone; support must approve it explicitly
- the security review verifies that teacher and student sessions are denied from these admin-only recovery routes

Current outcome:
